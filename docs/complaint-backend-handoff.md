# 불편 접수 서버 연동 인수인계

## 현재 상태

불편 접수 화면은 프론트엔드 동작만 구현되어 있습니다. 현재 유효한 폼을 제출하면 `ComplaintForm.tsx`의 `handleValidSubmit`이 성공 팝업을 바로 열며, 서버로 접수 내용을 전송하거나 저장하지 않습니다. 팝업을 닫으면 작성값과 첨부파일이 초기화되므로 서버 연동 전에는 실제 접수 데이터가 남지 않습니다.

휴대폰 인증도 실제 API가 아니라 임시 구현입니다.

- 인증 요청 함수: `apps/user/app/(site)/complaint/phoneVerification.ts`
- 현재 응답 상태: `not-configured`
- 임시 인증번호: `123456`
- 임시 인증번호 상수: `COMPLAINT_TEMP_VERIFICATION_CODE`

## 연동 시 지켜야 할 화면 흐름

1. React Hook Form 검증을 통과한 값을 서버로 전송합니다.
2. 서버가 접수를 저장하고 성공 응답을 반환한 뒤에만 성공 팝업을 엽니다.
3. 요청 중에는 중복 제출을 막고 접수 버튼에 진행 상태를 표시합니다.
4. 서버 오류가 발생하면 성공 팝업을 열지 않고 작성값과 첨부파일을 유지합니다.
5. 성공 팝업이 닫힐 때만 폼, 첨부파일, 인증 상태를 초기화합니다.

## 요청 데이터

서버 API 경로와 전송 형식은 서버 구현 시 확정해야 합니다. 첨부파일이 있으므로 `multipart/form-data` 사용을 권장합니다.

| 필드                              | 형식    | 필수   | 설명                           |
| --------------------------------- | ------- | ------ | ------------------------------ |
| `name`                            | string  | 예     | 접수자 이름                    |
| `email`                           | string  | 예     | 답변 받을 이메일               |
| `phone`                           | string  | 예     | 하이픈 없는 국내 이동전화 번호 |
| `service`                         | string  | 예     | 이용 서비스 제목               |
| `complaintType`                   | string  | 예     | 불편 유형 제목                 |
| `detail`                          | string  | 예     | 상세 내용                      |
| `privacy`                         | boolean | 예     | 개인정보 수집 및 이용 동의     |
| `verificationCode` 또는 인증 토큰 | string  | 예     | 서버 인증 방식 확정 후 교체    |
| `attachments`                     | file[]  | 아니요 | 최대 10개, 파일당 최대 50MB    |

성공 응답에는 추적 가능한 `complaintId`와 `createdAt`을 포함하는 것을 권장합니다. 프론트엔드는 HTTP 성공 여부뿐 아니라 이 식별자가 존재하는지 확인한 뒤 팝업을 열 수 있습니다.

## 서버 검증

클라이언트 검증은 사용자 경험을 위한 것이므로 서버에서 동일한 규칙을 다시 검증해야 합니다.

- 필수값과 개인정보 동의 여부
- 이메일 형식
- 국내 이동전화 번호 형식
- 서비스 및 불편 유형 허용 목록
- 상세 내용 길이와 금지 콘텐츠 정책
- 첨부파일 개수, 크기, MIME 타입 및 악성 파일 검사
- 실제 휴대폰 인증 완료 여부

## 프론트엔드 교체 지점

- `handleValidSubmit`: 서버 접수 요청을 호출하는 비동기 함수로 변경
- `requestPhoneVerification`: 실제 인증번호 발송 API로 변경
- `COMPLAINT_TEMP_VERIFICATION_CODE`: 제거하고 서버 검증 결과 또는 인증 토큰 사용
- `phoneVerificationResult`: 요청 성공, 실패, 만료 상태를 화면에 반영

서버 연동이 완료되기 전까지 현재 성공 팝업은 실제 접수 완료를 의미하지 않습니다.
